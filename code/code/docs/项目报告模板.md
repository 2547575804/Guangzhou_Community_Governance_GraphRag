# 广州市社区治理知识图谱与智能问答系统
## 项目报告

---

## 摘要

本项目针对广州市社区治理领域数据分散、检索效率低、缺乏智能分析等问题，设计并实现了一个基于Neo4j知识图谱与DeepSeek大语言模型的智能问答系统。系统整合了12345热线案例、法律文书和新闻报道三类数据源，构建了包含9类节点、8类关系的社区治理知识图谱，实现了跨数据源的语义关联。问答系统采用三分支并行架构（知识库检索、图谱推理、直接LLM应答），通过意图识别和上下文工程整合，为用户提供结构化的智能回答。实验结果表明，系统能够有效支持社区治理决策，提升案例检索效率和关联分析能力。

**关键词**：知识图谱、Neo4j、智能问答、社区治理、大语言模型、DeepSeek

---

## 第一章 项目背景与需求分析

### 1.1 项目背景

社区治理是城市基层管理的重要组成部分，涉及12345热线投诉处理、矛盾纠纷调解、公共服务提供等多个领域。当前社区治理面临以下挑战：

**（1）数据分散存储**
社区治理相关数据分散在多个系统中：12345热线平台存储投诉工单、法院系统存储法律文书、新闻媒体存储报道文章。这些数据缺乏统一整合，难以进行跨源分析。

**（2）检索效率低下**
传统的关键词检索只能匹配文本内容，无法发现案例之间的深层关联，如同一部门处理的不同案例、同一冲突类型的相关法律依据等。

**（3）缺乏智能分析**
基层工作人员在处理新案例时，需要人工查阅历史案例和法律条文，缺乏智能推荐和关系推理支持，决策响应速度慢。

### 1.2 需求分析

基于上述背景，本项目提出以下需求：

**（1）数据整合需求**
需要将12345热线案例、法律文书、新闻报道三类数据整合到统一的知识图谱中，建立实体之间的语义关联。

**（2）智能检索需求**
支持自然语言查询，能够根据用户问题自动识别意图，从知识图谱和知识库中检索相关信息。

**（3）关系推理需求**
能够分析案例之间的关联关系，如部门-案例、冲突类型-法律、跨域案例关联等，支持决策分析。

**（4）可视化需求**
提供知识图谱可视化界面，展示节点和关系的分布情况，便于用户直观理解数据结构。

### 1.3 项目目标

本项目旨在实现以下目标：

1. 构建社区治理领域知识图谱，整合多源异构数据
2. 实现跨数据源的语义关联（热线-法律文书-新闻）
3. 开发智能问答系统，支持自然语言交互
4. 提供图谱可视化与关系推理功能

## 第二章 技术方案设计

### 2.1 系统架构设计

系统采用分层架构设计，包括数据层、服务层和应用层：

**（1）数据层**
- CSV数据文件：存储原始案例数据
- Neo4j知识图谱：存储节点和关系
- 配置文件：存储系统参数

**（2）服务层**
- 意图识别服务（IntentService）：识别用户问题意图
- 知识库检索服务（KBService）：全文检索CSV数据
- 图谱推理服务（Neo4jService）：图谱查询与推理
- LLM服务（LLMService）：调用DeepSeek API
- 问答编排服务（QAService）：整合各分支回答

**（3）应用层**
- Flask Web服务：提供HTTP API
- 前端界面：用户交互界面

### 2.2 知识图谱建模

#### 2.2.1 节点类型设计

知识图谱包含以下9类节点：

| 节点类型 | 属性 | 说明 | 数量 |
|---------|------|------|------|
| WorkOrder | id, title, content, conflict_type, process, result | 12345热线工单 | 100 |
| CourtCase | id, case_number, dispute_type, key_facts, applicable_law, judgment_result | 法律文书 | 100 |
| News | id, title, content, conflict_type, solution, departments | 新闻报道 | 100 |
| Department | name | 处置部门 | 330 |
| City | name | 城市 | 1 |
| ConflictType | name | 冲突类型 | 103 |
| DisputeType | name | 纠纷类型 | 43 |
| Law | name | 法律条文 | 35 |
| Media | name | 新闻媒体 | 9 |

#### 2.2.2 关系类型设计

知识图谱包含以下8类关系：

| 关系类型 | 起点 | 终点 | 说明 |
|---------|------|------|------|
| LOCATED_IN | WorkOrder/CourtCase/News | City | 所在城市 |
| HANDLED_BY | WorkOrder | Department | 处置部门 |
| HAS_CONFLICT_TYPE | WorkOrder/News | ConflictType | 冲突类型 |
| HAS_DISPUTE_TYPE | CourtCase | DisputeType | 纪纷类型 |
| APPLIES_LAW | CourtCase | Law | 适用法律 |
| INVOLVES | News | Department | 涉及部门 |
| REPORTED_BY | News | Media | 报道媒体 |
| RELATED_COURTCASE | WorkOrder/News | CourtCase | 跨域关联 |

#### 2.2.3 跨域关联设计

为实现热线案例、法律文书、新闻报道三类数据的深度融合，设计了以下跨域关联规则：

**（1）同部门关联**
热线工单与新闻报道通过共同涉及的部门建立关联：
```
MATCH (w:WorkOrder)-[:HANDLED_BY]->(d:Department)<-[:INVOLVES]-(n:News)
MERGE (w)-[:RELATED_NEWS {relation: '同处置部门'}]->(n)
```

**（2）同冲突类型关联**
热线工单与新闻报道通过共同的冲突类型建立关联：
```
MATCH (w:WorkOrder)-[:HAS_CONFLICT_TYPE]->(t:ConflictType)<-[:HAS_CONFLICT_TYPE]-(n:News)
MERGE (w)-[:RELATED_NEWS {relation: '同冲突类型'}]->(n)
```

**（3）语义匹配关联**
热线工单/新闻与法律文书通过冲突类型和纠纷类型的语义匹配建立关联：
```
MATCH (w:WorkOrder)-[:HAS_CONFLICT_TYPE]->(ct:ConflictType)
MATCH (cc:CourtCase)-[:HAS_DISPUTE_TYPE]->(dt:DisputeType)
WHERE ct.name CONTAINS dt.name OR dt.name CONTAINS ct.name
MERGE (w)-[:RELATED_COURTCASE {relation: '冲突/纠纷类型相关'}]->(cc)
```

### 2.3 问答系统设计

#### 2.3.1 三分支并行架构

问答系统采用三分支并行处理架构：

**分支1：知识库检索（KBService）**
- 基于关键词从CSV文件中检索相关案例
- 返回案例标题、内容、来源类型
- 适用于具体案例查询

**分支2：图谱推理（Neo4jService）**
- 关键词模糊检索图谱节点
- 关系推理查询（部门-案例、法律-案例）
- LLM生成Cypher查询
- 适用于关系分析类问题

**分支3：直接LLM应答（LLMService）**
- 直接调用DeepSeek API
- 适用于通用概念解释、闲聊类问题

#### 2.3.2 意图识别设计

意图识别采用规则引擎与LLM双重识别策略：

**（1）规则引擎识别**
基于关键词匹配识别意图：
- 图谱推理关键词：关系、关联、哪些部门、涉及、适用法律等
- 知识库检索关键词：案例、工单、判决、新闻、怎么处理等
- 直接LLM关键词：你好、什么是、介绍、定义等

**（2）LLM识别**
当规则引擎置信度低于0.85时，调用DeepSeek API进行意图识别。

**（3）意图类型**
- knowledge_base：知识库查询
- graph_reasoning：推理查询
- direct_llm：通用问答

#### 2.3.3 上下文工程

问答系统通过上下文工程整合三个分支的回答：

**整合策略：**
- 根据意图识别结果确定主要参考分支
- 去重、消歧，避免重复罗列
- 优先采用与问题最相关的信息
- 案例引用注明来源
- 结构化输出：结论 → 依据 → 建议

```python
class KnowledgeGraphBuilder:
    def build(self, clear: bool = True):
        if clear:
            self.clear_database()  # 清空现有数据
        self.create_constraints()  # 创建唯一性约束
        self.import_workorders()   # 导入热线案例
        self.import_court_cases()  # 导入法律文书
        self.import_news()         # 导入新闻报道
        self.create_cross_domain_relations()  # 创建跨域关联
        self.print_statistics()    # 输出统计信息
```

### 3.2 服务层实现

#### 3.2.1 Neo4j服务（neo4j_service.py）

```python
class Neo4jService:
    def reasoning_search(self, question: str, keywords: list[str]) -> dict:
        """综合图谱推理检索"""
        node_results = self.search_by_keyword(keywords)  # 节点检索
        relation_results = self.query_relations(keywords)  # 关系推理
        cypher, cypher_results = self.generate_cypher_and_query(question)  # LLM生成查询
        return {
            "node_results": node_results,
            "relation_results": relation_results,
            "cypher": cypher,
            "cypher_results": cypher_results,
        }
```

#### 3.2.2 LLM服务（llm_service.py）

```python
class LLMService:
    def chat(self, messages: list[dict], temperature: float = 0.3) -> str:
        """调用DeepSeek API"""
        url = f"{self.base_url}/v1/chat/completions"
        payload = {
            "model": self.model,
            "messages": messages,
            "temperature": temperature,
        }
        response = requests.post(url, headers=headers, json=payload)
        return response.json()["choices"][0]["message"]["content"]
```

```python
class QAService:
    def answer(self, question: str) -> dict:

        branch_kb = self._branch_knowledge_base(question)  # 分支1
        branch_graph = self._branch_graph_reasoning(question, keywords)  # 分支2
        branch_direct = self._branch_direct_llm(question)  # 分支3
        
        final_answer = self._integrate_response(...)  # 整合回答
        return {"question": question, "answer": final_answer, ...}
```

### 3.3 应用层实现

#### 3.3.1 Flask API（app.py）

```python
@app.route("/api/chat", methods=["POST"])
def chat():
    """问答API"""
    question = request.get_json().get("question")
    result = qa_service.answer(question)
    return jsonify(result)

@app.route("/api/kg/stats", methods=["GET"])
def kg_stats():
    """图谱统计API"""
    node_stats = neo4j_service._run("MATCH (n) RETURN labels(n)[0], count(n)")
    rel_stats = neo4j_service._run("MATCH ()-[r]->() RETURN type(r), count(r)")
    return jsonify({"nodes": node_stats, "relationships": rel_stats})
```

## 第五章 总结与展望

### 5.1 项目总结

本项目成功实现了广州市社区治理知识图谱与智能问答系统，主要成果包括：

**（1）知识图谱构建**
- 整合了12345热线案例、法律文书、新闻报道三类数据
- 构建了9类节点、8类关系的知识图谱
- 实现了跨数据源的语义关联

**（2）智能问答系统**
- 实现了三分支并行处理架构
- 实现了意图识别与上下文工程整合
- 支持自然语言交互

**（3）可视化与推理**
- 提供Neo4j浏览器可视化界面
- 支持关系推理查询
- 支持LLM生成Cypher查询

### 5.2 技术亮点

**（1）跨域语义关联**
通过冲突类型和纠纷类型的语义匹配，实现了热线案例与法律文书的精准关联，避免了简单的"同城市"关联带来的噪声。

**（2）三分支并行架构**
知识库检索、图谱推理、直接LLM三个分支并行处理，通过意图识别和上下文工程整合，提高了回答的准确性和完整性。

**（3）智能意图识别**
规则引擎与LLM双重识别策略，在保证效率的同时提高了识别准确率。

### 5.3 未来展望

**功能扩展方向：**
1. 数据规模扩展：接入更多案例数据，扩大图谱覆盖范围
2. 新数据源接入：整合政策文件、法规数据库等
3. 多模型支持：接入更多大语言模型，提高回答质量

**应用拓展方向：**
1. 政务系统集成：与现有政务平台对接，提供决策支持
2. 移动端应用：开发移动端问答应用，方便基层工作人员使用
3. 实时舆情预警：基于新闻报道数据，实现舆情监测和预警
